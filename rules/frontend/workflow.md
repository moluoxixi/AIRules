# Frontend Workflow Rules

适用于前端页面、组件和交互任务的标准流程规则。

## Trigger

当任务主要目标是以下任一场景时，必须遵循本规则：

- 写页面
- 改页面
- 补页面交互
- 页面联调
- 页面与接口联动调试

## Implementation

页面任务默认按以下顺序推进：

1. 先识别当前技术栈，并遵循对应栈规则实现
2. 按页面层、组件层、复用逻辑层的职责边界完成代码
3. 为导出组件、composables、hooks 与复杂 util 补齐注释，遵循 `rules/frontend/jsdoc.md`

## MCP 验证规则

- 页面实现完成后，必须询问用户是否需要进行 MCP 验证
- 如果用户同意，使用 MCP 验证关键用户路径、用户可见行为以及 loading / empty / error / success 状态
- MCP 验证时，使用 `webapp-testing` (vendor) 进行 Playwright 浏览器测试
- 如果用户拒绝或当前环境无法执行 MCP，必须在交付说明中明确标注"未执行 MCP 验证"

## Uncertainty Handling

当前端行为、接口结构、交互结果或页面状态不确定时，优先顺序如下：

1. 先通过 MCP 调试、查看真实页面行为或真实接口返回
2. 再读取现有代码、调用方和上下文
3. 仍无法确定时，再向用户提问

## Avoid

- 不因信息不完整而先写大段兼容逻辑
- 不在未验证真实行为前假设接口返回结构
- 不把“可能需要兼容”当作默认实现前提
