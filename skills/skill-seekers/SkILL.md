## 零门槛入门 Skills

### 🤔 Skill 是啥？

**说人话：Skill 就是给 AI 的"小抄"。**

你考试的时候，如果允许带一张小抄，上面写着公式、要点、注意事项，你是不是做题又快又准？

Skill 就是这张小抄 —— 只不过用它的不是你，是你的 AI 助手 🤖

| | 没有 Skill 的 AI 😐 | 有 Skill 的 AI 🧠 |
|---|---|---|
| **你说** | "帮我查一下 Aone 上的 Bug" | "帮我查一下 Aone 上的 Bug" |
| **AI 回答** | "Aone 是什么？我不太了解你们的系统……" | "好的！我来查 Aone，找到 5 个 Bug，要我逐个分析吗？" |
| **你的感受** | 😤 还不如自己查 | 😍 真香！ |

> 💬 **一个比喻**：没有 Skill 的 AI = 聪明但啥也不懂的应届生；有了 Skill 的 AI = 干了三年的老员工，门儿清。

### 📦 Skill 长啥样？怎么让 AI 用上？

别被名字吓到。Skill 就是一个**文件夹**，里面放了几个文件：

```
my-skill/
├── 📄 SKILL.md       ← 核心！告诉 AI "你会什么、怎么做"
├── 📚 references/    ← 详细资料，AI 需要时才翻
├── 🔧 scripts/       ← 可以直接跑的脚本
└── 📎 assets/        ← 模板、配置文件等
```

**怎么让 AI 用上这个 Skill？** 把这个文件夹放到你的项目目录里就行了。AI 助手（比如 Cursor、Claude）会**自动读取**项目里的 Skill 文件，不需要你手动"安装"什么东西。就像你把一份参考资料放到桌上，AI 自己就会翻开来看。

**举个真实例子** —— 我们团队的 Bug 修复 Skill 🐛：

把这个 Skill 放到项目里后，你只需要说 **"帮我看看最近有什么 Bug"**，AI 就会自动查 Aone、列清单、分析方案、等你确认后才改状态。

**你不用记任何 API 参数，不用翻任何文档，张嘴就行。**

### 🎨 神器一：skill-creator-pro — 你动嘴，AI 动手

#### 它是什么？

skill-creator-pro 本身也是一个 Skill。把它放到项目里后，AI 就自动学会了"怎么创建高质量 Skill"。

**你不需要学任何规范**，只要用大白话描述你的工作流程，AI 就会自动帮你生成一个标准的 Skill 包。格式怎么写、内容怎么组织、文件怎么放——全部由 AI 搞定。

而且它不只是"创建"——还能**自动测试、评分、优化**你的 Skill，确保质量过关。

#### 它能帮你做什么？

- ✅ **理解你的场景** — 你说"我想做个发布流程的 Skill"，AI 会追问细节，帮你梳理清楚
- ✅ **自动生成完整的 Skill 包** — `SKILL.md` 核心文案 + 参考资料 + 脚本 + 资产，一次性全部生成
- ✅ **设置安全规则** — 比如"正式发布前必须通过预发验证"，AI 会自动加上护栏
- ✅ **渐进式披露** — AI 会自动把内容分层：核心指令放 `SKILL.md`，详细资料放 `references/`，避免一次性塞太多内容撑爆上下文
- ✅ **自动测试评估** — 生成测试用例，自动跑"带技能 vs 不带技能"的 A/B 对比，用 AI 评分打分
- ✅ **基准测试** — 多轮运行获得统计显著性，方差分析确认改进是否真的有效
- ✅ **描述优化** — 自动优化技能的触发描述，提高 AI 识别和使用技能的准确率
- ✅ **持续优化** — 用了一段时间觉得哪里不好？跟 AI 说一声，它帮你改

#### 它生成的 Skill 包长什么样？

```
my-skill/
├── SKILL.md           ← 核心文案：告诉 AI "你会什么、怎么做"
├── references/        ← 详细参考资料（AI 需要时才读取，节省上下文）
│   ├── 发布前检查清单.md
│   └── 常见问题排查.md
├── scripts/           ← 可直接执行的脚本
│   └── 一键部署.sh
└── assets/            ← 模板、配置等输出文件
    └── 配置模板.yaml
```

> 💡 **核心是 `SKILL.md`**，其他都是辅助。AI 会先读 `SKILL.md` 了解整体流程，需要细节时再去翻 `references/` 里的资料。

| 没有 skill-creator-pro 😰 | 有 skill-creator-pro 🎉 |
|---|---|
| 要学 Skill 的文件格式和规范 | 不用学，AI 都知道 |
| 要自己组织内容结构 | AI 自动按最佳实践组织 |
| 要自己写脚本和配置 | AI 帮你生成 |
| 创建完不知道效果好不好 | 自动测试 + 评分 + 优化 |
| 创建一个 Skill 要半天 | 聊几句话就搞定 |

> 🔑 **一句话总结**：skill-creator-pro 把"创建 Skill"从"需要学习规范的技术活"变成了"跟 AI 聊天"，还能自动帮你测试和优化质量。

### 🔍 神器二：skill-seekers — 已有知识一键变 Skill

#### 它是什么？

skill-seekers 也是一个 Skill。它解决的是一个很实际的问题：**你想让 AI 学的知识，其实大部分已经存在了**——在文档网站上、在 GitHub 上、在 PDF 里、在视频里……

skill-seekers 能把这些**已有的知识自动抓取过来，变成标准的 Skill 包**。你不用手动复制粘贴，不用自己整理格式。

生成的 Skill 包和 skill-creator-pro 创建的结构一样——核心是 `SKILL.md`，加上 `references/` 里的详细资料。放到项目里就能用。

### 🧠 最佳实践：上下文工程

不管是用 skill-creator 从零创建，还是用 skill-seekers 抓取已有知识，最终生成的都是 Skill。那 Skill 到底是什么？怎么写才算好？这就要聊一个关键概念——**上下文工程**（Context Engineering）。

#### Skill 的本质是什么？

**Skill 本质上就是要加载到 AI 上下文窗口里的提示词（Prompt）。**

说白了，`SKILL.md` 就是一份精心编写的 Markdown 文档，告诉 AI："你现在是一个懂 XXX 的专家，按照以下流程干活。" AI 读了这份文档，就"学会"了这个技能。

#### 什么是上下文窗口？

**AI 大模型有一个"上下文窗口"**，你可以把它想象成 AI 的"工作台"。AI 每次干活时，工作台上能放的东西是有限的——你的对话记录、系统指令、代码文件……全都要挤在这张桌子上。

```
┌─────────────────────────────────────────────┐
│              AI 的上下文窗口（工作台）          │
│                                             │
│  📋 系统指令    💬 对话历史    📂 代码文件     │
│  📄 Skill A    📄 Skill B    📄 参考资料     │
│                                             │
│  ⚠️ 桌子就这么大，放太多东西会挤不下！         │
└─────────────────────────────────────────────┘
```

**上下文工程就是：精心管理这张工作台上放什么、什么时候放、放多少。**

- 放太少 → AI 不知道怎么干活（"Aone 是什么？"）
- 放太多 → AI 被信息淹没，反而变笨（关键信息被稀释）
- 放得刚好 → AI 又快又准 ✅

#### 所以 Skill 怎么写才算好？

Skill 的设计核心就是**上下文工程**——用最少的 token 传递最多的有效信息。好的 Skill 不是写得多，而是**写得精**：

| 设计原则 | 做法 | 为什么 |
|---|---|---|
| **精简核心** | `SKILL.md` 控制在 **500 行以内** | 上下文窗口是公共资源，不能一个 Skill 就占满 |
| **按需加载** | 详细资料放 `references/`，AI 需要时才读 | 不用的信息不占工作台空间 |
| **按领域拆分** | 参考资料按功能拆成多个小文件 | AI 只读需要的那个文件，不会全部加载 |
| **避免重复** | 同一信息只存在一个地方 | 重复信息浪费宝贵的上下文空间 |

Skill 使用**三级加载机制**，像翻书一样按需读取：

```
第 1 级：元数据（name + description）
         → 始终在上下文中，~100 字
         → AI 靠这个判断"要不要用这个 Skill"

第 2 级：SKILL.md 正文
         → 技能被触发时加载，<500 行
         → 核心工作流程和指令

第 3 级：references/ 参考资料
         → AI 判断需要时才读取，无大小限制
         → 详细的规则、模板、API 文档等
```

> 💡 **一个比喻**：`SKILL.md` 是**目录**，`references/` 是**各章节**。AI 先看目录知道大纲，需要哪章翻哪章——而不是把整本书一次性塞进脑子里。

> 🔑 **一句话总结**：好的 Skill = 用最少的字，让 AI 获得最多的能力。不管你用 skill-creator-pro 还是 skill-seekers 创建 Skill，它们都会自动帮你遵循这些最佳实践。


## 数字员工

### 🤖 什么是数字员工？

前面说的 Skill，是给 AI 的"小抄"。**数字员工就是装了一堆 Skill 的 AI 机器人**——它掌握了一个岗位所需的全部知识和流程，能像真人同事一样干活。

Skill 决定了数字员工**能做什么**。你给它装上"发布上线 Skill"，它就会发布；装上"Bug 修复 Skill"，它就会修 Bug。Skills 越多，数字员工越强。

## 共建

### 📦 Skills 仓库

所有 Skill 统一管理在一个 Git 仓库里：

> 🔗 `http://gitlab.alibaba-inc.com/wlkj/skills.git`

这个仓库不是某个人的项目。**每个 Skill 都有自己的作者，每个贡献者的名字都在 Git 提交记录里。**

### 为什么是共建？

数字员工的能力来自 Skills，Skills 来自团队每个人的经验。这不是一个"某某开发了一个工具，大家来用"的模式——而是**所有人都是源码贡献者**。

- 你懂发布流程 → 你贡献发布 Skill → 你是发布能力的作者
- 他懂 Bug 修复 → 他贡献修复 Skill → 他是修复能力的作者
- 她懂性能优化 → 她贡献优化 Skill → 她是优化能力的作者

数字员工的每一项能力，都能追溯到具体的贡献者。**荣誉属于每一个参与的人，而不是某一个开发者。**

### 怎么参与？

1. **克隆仓库**：`git clone http://gitlab.alibaba-inc.com/wlkj/skills.git`
2. **创建 Skill**：用 skill-creator-pro 或 skill-seekers（前面介绍的两个神器）
3. **提交 MR**：把你的 Skill 提交到仓库
4. **合并上线**：Review 通过后，数字员工就自动获得了你贡献的新能力

> 🎯 **现在就开始**：找一个你最熟悉的工作流程，把它变成 Skill，提交到 Skills 仓库。
